Business Problem

Azenta Life Sciences manages biorepository services for pharmaceutical clients (BMS, Sanofi, etc.) under Master Service Agreements. Each MSA contains dense pricing tables covering:

  • Storage Pricing — 6 temperature conditions × 11 sample-size columns × 5–6 quantity tiers per page
  • Management Fees — Project initiation, sample administration, lab services, professional services, additional services
  • Multi-currency — USD (North America), EUR (Griesheim), SGD (Singapore)
  • Multi-region — North America, Europe, Singapore, Beijing

Manually extracting pricing from ~600 MSA documents is error-prone and time-consuming. This pipeline automates extraction with AI, enabling systematic pricing analysis and contract compliance validation.

Parse-Cache Pattern: AI_PARSE_DOCUMENT is billed per page (Arctic-TILT model). By caching parsed text in PARSED_CONTRACTS_TEXT, each PDF is parsed exactly once. Both Pipeline A and B read from the same cache, eliminating re-parsing costs when switching models or re-running extractions.

Why AI_PARSE_DOCUMENT + Separate Extraction?

Snowflake's recommended best practice for complex documents is the two-stage approach:

StageFunctionPurposeModel
Stage 1AI_PARSE_DOCUMENTLayout-aware PDF → Markdown, preserves table structureArctic-TILT (90%+ table accuracy)
Stage 2AAI_EXTRACTSchema-defined extraction with parallel array outputarctic-extract (128K context)
Stage 2BAI_COMPLETEPrompt-based extraction with JSON response_formatConfigurable (128K–200K context)
✓
Why not AI_EXTRACT directly on the file? AI_EXTRACT can operate on raw files but uses OCR-level extraction without layout intelligence. For dense pricing matrices with complex column alignment, AI_PARSE_DOCUMENT(mode='LAYOUT') with Arctic-TILT produces significantly higher accuracy Markdown before extraction.

Pipeline A — AI_PARSE_DOCUMENT + AI_EXTRACT

Schema-driven extraction using arctic-extract model with parallel array output

How Pipeline A Works

Step 1 — Parse with Cache Check

-- Cache check: only parse if not already in table
LET already_parsed NUMBER;
SELECT COUNT(*) INTO :already_parsed
FROM SRS_AI_CONTRACTS.RAW_DATA.PARSED_CONTRACTS_TEXT
WHERE FILENAME = :P_FILENAME;

IF (:already_parsed = 0) THEN
    LET parsed VARIANT;
    SELECT AI_PARSE_DOCUMENT(
        TO_FILE('@SRS_AI_CONTRACTS.RAW_DATA.CONTRACT_STAGE', :P_FILENAME),
        {'mode': 'LAYOUT', 'page_split': TRUE}
    ) INTO :parsed;

    -- Insert one row per page into cache
    INSERT INTO PARSED_CONTRACTS_TEXT (FILENAME, CUSTOMER_ID, PAGE_INDEX, PAGE_TEXT, TOTAL_PAGES)
    SELECT :P_FILENAME, :P_CUSTOMER_ID,
           p.index::NUMBER,
           p.value:content::VARCHAR,
           :parsed:metadata:pageCount::NUMBER
    FROM TABLE(FLATTEN(input => :parsed:pages)) p;
END IF;

Step 2 — AI_EXTRACT with Schema Definition

SELECT
    p.FILENAME,
    AI_EXTRACT(
        p.PAGE_TEXT,
        responseFormat => {
            'schema': {
                'type': 'object',
                'properties': {
                    'storage_rows': {
                        'description': 'All rows from storage pricing tables...',
                        'type': 'object',
                        'column_ordering': ['temperature', 'sample_size_label', ...],
                        'properties': {
                            'temperature':       {'type': 'array'},
                            'sample_size_label': {'type': 'array'},
                            'qty_tier_label':    {'type': 'array'},
                            'price_usd':         {'type': 'array'},
                            'price_eur':         {'type': 'array'},
                            'region':            {'type': 'array'}
                        }
                    }
                }
            }
        }
    ):response AS extraction
FROM PARSED_CONTRACTS_TEXT p
WHERE p.FILENAME = :P_FILENAME
  AND (LOWER(p.PAGE_TEXT) LIKE '%storage%'
       OR LOWER(p.PAGE_TEXT) LIKE '%ambient%' ...)

Step 3 — ARRAY_GENERATE_RANGE Flatten Pattern

AI_EXTRACT returns parallel arrays for each column. This pattern converts parallel arrays into individual rows:

-- Flatten parallel arrays: temperature[], price_usd[], qty_tier[] → rows
flattened AS (
    SELECT
        pe.FILENAME,
        pe.extraction:storage_rows:temperature[idx.value]::VARCHAR       AS temperature,
        pe.extraction:storage_rows:sample_size_label[idx.value]::VARCHAR AS sample_size,
        pe.extraction:storage_rows:price_usd[idx.value]::VARCHAR         AS price_usd
    FROM page_extractions pe,
    TABLE(FLATTEN(
        input => ARRAY_GENERATE_RANGE(
            0, ARRAY_SIZE(pe.extraction:storage_rows:temperature)
        ),
        MODE => 'ARRAY'
    )) AS idx
)
✓
Why ARRAY_GENERATE_RANGE? AI_EXTRACT's parallel-array schema (all columns as equal-length arrays) is the most token-efficient format. ARRAY_GENERATE_RANGE(0, N) generates index positions 0 to N-1, which are then used to index into all parallel arrays simultaneously — one row per index position.
Section 5

Pipeline B — AI_PARSE_DOCUMENT + AI_COMPLETE

Prompt-based extraction with swappable LLM models and JSON response enforcement

📄
File: pipelines/13_pipeline_b_ai_parse_complete.sql  |  Procedure: SRS_AI_CONTRACTS.RAW_DATA.EXTRACT_PIPELINE_B(P_FILENAME, P_CUSTOMER_ID, P_MODEL DEFAULT 'mistral-large2')

Key Differences from Pipeline A

Pipeline A — AI_EXTRACT

  • Schema-defined via JSON object spec
  • Model: always arctic-extract
  • Output: parallel arrays → ARRAY_GENERATE_RANGE
  • Output cap: 4,096 tokens per table question
  • Best for: consistent schema, high throughput

Pipeline B — AI_COMPLETE

  • Prompt-based with detailed instructions
  • Model: parameter — swap at call time
  • Output: JSON object-per-row → LATERAL FLATTEN
  • Output: up to model's output token limit
  • Best for: complex reasoning, model comparison

Model Guard — Blocking snowflake-arctic

-- GUARD: Block snowflake-arctic — 4,096 token context window too small
IF (:P_MODEL = 'snowflake-arctic') THEN
    RETURN 'ERROR: snowflake-arctic has a 4,096 token total context window '
        || '(input + output combined), which is insufficient for dense MSA pricing tables. '
        || 'Use mistral-large2, llama3.3-70b, claude-4-sonnet, or llama4-maverick instead.';
END IF;

AI_COMPLETE with JSON Response Format

TRY_PARSE_JSON(
    AI_COMPLETE(
        :P_MODEL,
        'You are extracting data from an Azenta Life Sciences Rate Card page. '
        || 'Extract ALL storage pricing rows into JSON with key "storage_rows". '
        || '"storage_rows" must be an array where each element has EXACTLY these keys: '
        || 'temperature, sample_size_label, min_sample_ml, max_sample_ml, '
        || 'cubic_foot_max, qty_tier_label, min_qty, max_qty, '
        || 'price_usd, price_eur, price_sgd, region. '
        || 'If no storage pricing on this page, return {"storage_rows": []}. '
        || 'Return ONLY valid JSON. No explanation, no markdown fences.'
        || CHR(10) || 'PAGE CONTENT:' || CHR(10) || p.PAGE_TEXT,
        {'response_format': {'type': 'json'}}  -- enforces structured output
    )
) AS result

Pipeline B Output Flatten — LATERAL FLATTEN on object-per-row JSON

-- Pipeline B returns array of objects (one per row) — use LATERAL FLATTEN
FROM page_completions pc,
LATERAL FLATTEN(input => pc.result:storage_rows) row

SELECT
    row.value:temperature::VARCHAR       AS TEMPERATURE,
    row.value:sample_size_label::VARCHAR AS SAMPLE_SIZE,
    row.value:price_usd::VARCHAR         AS PRICE

Section 6

Token Limit Strategy

How the pipeline avoids the AI_EXTRACT 4,096 token output cap and AI_COMPLETE context limits

Understanding the Token Limits

⚠
Common Misconception: The 4,096 token limit is NOT a global context window limit. It is the output token cap per table question in AI_EXTRACT. A dense pricing page with 6 temperature rows × 11 sample sizes × 6 quantity tiers = 396 cells can overflow this cap when extracted from the full document at once.
Limit TypeValueApplies ToSolution
AI_EXTRACT output per table question4,096 tokensPipeline Apage_split=TRUE — extract per page
arctic-extract context window128K tokensPipeline ANo action needed — well within limit
snowflake-arctic total context4,096 tokensPipeline B guardBlocked via explicit RETURN error
mistral-large2 / llama3.3 context128K tokensPipeline BPer-page processing, typically <2K tokens input
claude-4-sonnet context200K tokensPipeline BCould handle full document, but page-split used for consistency

Why page_split=TRUE Solves the Problem

Without page_split (problematic)

  • Full 18-page Sanofi PDF = ~36,000 tokens input
  • AI_EXTRACT attempts to extract all 396 cells at once
  • Output exceeds 4,096 token cap → truncation
  • Missing rows, corrupted arrays

With page_split=TRUE (correct)

  • Each page = ~1,000–3,000 tokens input
  • Each page has at most 1–2 temperature blocks
  • Output per page ≈ 200–800 tokens (well under 4,096)
  • Complete extraction across all pages

Keyword Pre-filtering

Before calling AI functions, both pipelines filter pages with SQL LIKE conditions:

WHERE p.FILENAME = :P_FILENAME
  AND (
      LOWER(p.PAGE_TEXT) LIKE '%storage%'
      OR LOWER(p.PAGE_TEXT) LIKE '%ambient%'
      OR LOWER(p.PAGE_TEXT) LIKE '%refrigerat%'
      OR LOWER(p.PAGE_TEXT) LIKE '%nitrogen%'
      OR LOWER(p.PAGE_TEXT) LIKE '%-80%'
  )

For a 13-page BMS document, this typically reduces AI calls from 13 to 4–6 pricing-relevant pages, cutting compute costs by ~50–60%.

Three Validation Views

View NameSchemaPurpose
EXTRACTION_VALIDATION_PRICINGSRS_AI_CONTRACTS.ANALYTICSRow-level storage pricing comparison vs ground truth
EXTRACTION_VALIDATION_MGMTSRS_AI_CONTRACTS.ANALYTICSRow-level management fee comparison vs ground truth
EXTRACTION_VALIDATION_SUMMARYSRS_AI_CONTRACTS.ANALYTICSAggregate accuracy metrics per extraction method

Match Status Logic

StatusConditionMeaning
EXACT_MATCHextracted price = expected price (string)Perfect extraction
NEAR_MATCHprices within 1% after numeric parseRounding or formatting difference
WITHIN_1PCTABS(extracted - expected) / expected < 0.01Minor numeric deviation
MISMATCHBoth have values but differ > 1%Extraction error — needs review
NO_EXPECTEDExtracted row has no ground-truth matchHallucinated row or unlabeled data

Validation Summary Query Pattern

SELECT
    EXTRACTION_METHOD,
    COUNT(*) AS total_rows,
    SUM(CASE WHEN MATCH_STATUS IN ('EXACT_MATCH', 'NEAR_MATCH', 'WITHIN_1PCT') THEN 1 ELSE 0 END) AS matched_rows,
    ROUND(100.0 * SUM(CASE WHEN MATCH_STATUS IN ('EXACT_MATCH', ...) THEN 1 ELSE 0 END)
         / NULLIF(COUNT(*), 0), 2) AS accuracy_pct
FROM SRS_AI_CONTRACTS.ANALYTICS.EXTRACTION_VALIDATION_PRICING
GROUP BY EXTRACTION_METHOD
ORDER BY accuracy_pct DESC;

Batch Processing (600 MSAs)

-- Batch pattern: iterate over a list of filenames
DECLARE
    c CURSOR FOR
        SELECT FILENAME, CUSTOMER_ID FROM SRS_AI_CONTRACTS.RAW_DATA.CONTRACT_METADATA
        WHERE PROCESSED = FALSE;
BEGIN
    FOR rec IN c DO
        CALL SRS_AI_CONTRACTS.RAW_DATA.EXTRACT_PIPELINE_A(rec.FILENAME, rec.CUSTOMER_ID);
        UPDATE SRS_AI_CONTRACTS.RAW_DATA.CONTRACT_METADATA
            SET PROCESSED = TRUE WHERE FILENAME = rec.FILENAME;
    END FOR;
END;

⚠
Why is snowflake-arctic blocked? snowflake-arctic's total context window (input + output combined) is 4,096 tokens. A single pricing page from a Sanofi Rate Card is approximately 2,000–4,000 tokens. Even with page-splitting, the combined prompt + page text + JSON response would exceed this limit, causing truncated or empty responses. The guard in Pipeline B returns a descriptive error string instead of throwing an exception, enabling safe batch processing.

Recommendation for 600 MSA Scale

Phase 1: Bulk Extraction

Use Pipeline A (AI_EXTRACT) with arctic-extract — schema-based, deterministic, lower per-token cost. Extract all 600 MSAs once into the output tables.

Phase 2: Validation & Comparison

Run Pipeline B with llama3.3-70b on a 10% sample. Compare outputs in the Streamlit A vs B tab. Escalate to claude-4-sonnet for documents where Pipeline A shows low confidence.
